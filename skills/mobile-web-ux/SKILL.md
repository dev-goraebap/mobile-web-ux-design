# 모바일 웹 UX — 네이티브 앱처럼 만들기

웹 UI를 모바일에서 "앱 같다"고 느끼게 만드는 일은, 막상 들여다보면 성격이 다른 몇 개의 관심사를 쌓는 일이다. 이 스킬은 그 관심사를 네 개의 층으로 나누고, 각 층에서 내려야 하는 결정과 그 근거를 정리한다. 중요한 것은 특정 프레임워크의 문법이 아니라 이 층 모델과 결정이며, 그래서 이 레포의 Angular 구현은 가능한 여러 구현 중 하나일 뿐이다 — 원칙은 React나 Svelte로도 그대로 옮겨진다.

## 언제 쓰나

모바일에서 바텀시트·모달이나 끌어서 닫기·스와이프 같은 터치 제스처가 필요할 때, 화면 크기에 따라 레이아웃이 아니라 *표현 자체*가 바뀌어야 할 때(바텀시트↔모달, 탭바↔사이드 레일), safe-area·44px 터치 타깃·다크/라이트 같은 모바일 디테일을 챙겨야 할 때 이 스킬을 편다. 완성된 UI 라이브러리를 채택할지 디자인을 직접 소유할지 저울질하는 단계, 혹은 만든 웹앱을 설치·오프라인 가능한 PWA로 마감하려는 단계에도 해당한다.

## 네 개의 층

책임을 네 층으로 나눈다. 앞의 세 층은 "어떻게 느껴지는가"를, 마지막 한 층은 "어떻게 전달되는가"를 책임진다. 그래서 3 + 1이라고 부른다.

| 층 | 역할 | 무엇으로 |
| --- | --- | --- |
| 1. 디자인 시스템(소유) | 토큰 + 토큰으로 감싼 자체 컴포넌트 | Tailwind `@theme` + 자체 `ui/` |
| 2. 헤드리스 프리미티브 | 동작·접근성(스타일 없음) | Angular CDK / React Aria·Radix |
| 3. 모바일 UX | 적응형·safe-area·제스처 | CDK + CSS + GSAP |
| +. PWA 셸 | 설치·오프라인(전달) | service worker + manifest |

이렇게 나누면 한 가지가 분명해진다. PWA는 흔히 "모바일용 웹사이트"로 오해되지만, 실은 *설치·오프라인이라는 앱 능력*을 웹에 덧입히는 일이고, 앞의 세 층과는 성격이 다른 마감 작업이다. 네 층 중 프레임워크가 가장 덜 중요한 층이기도 하다.

## 첫 번째 결정 — 디자인 시스템을 소유할 것인가

이 결정을 공정하게 내리려면, 소유하지 *않는* 쪽의 이점을 먼저 인정하는 데서 시작하는 것이 옳다. Ionic이나 Material 같은 완성형 라이브러리는 스타일과 헤드리스 동작을, Ionic의 경우 모바일 UX의 상당 부분까지 한꺼번에 제공한다. 검증된 접근성과 빠른 출발이 그 보상이다. 작은 팀이 빨리 출시해야 한다면 이 길이 합리적이다.

대가는 제어권이다. 컴포넌트가 마크업을 캡슐화할수록 바깥에서 정의한 스타일이 안쪽까지 닿지 못하고, 디자인의 상당 부분을 라이브러리의 언어에 위임하게 된다. 결과물이 "그 라이브러리처럼" 보이는 것을 피하기 어렵다.

소유하는 쪽(YES)은 이 제어권을 되찾는 대신 시간과 전문성을 치른다. 다만 한 가지 오해를 짚어 둘 필요가 있다. 소유한다고 해서 헤드리스 프리미티브를 처음부터 짜는 것은 아니다. 포커스 트랩이나 오버레이 같은 동작은 여전히 CDK·Radix에서 *차용*하고, 그 위에 배선과 스타일만 얹는다. 즉 YES는 "외형을 소유하고 동작을 빌리는" 길이다. 이 스킬은 이 길을 따른다. 트레이드오프의 자세한 논의는 [references/01-design-system.md](references/01-design-system.md)에 있다.

## 쌓는 순서

순서에는 두 가지 이유가 있다. 첫째, 시각 토대를 먼저 깐다. 모든 시각 작업이 토큰에 의존하므로 색·간격·라운드·브레이크포인트·폰트를 `@theme`에 정의하는 것이 1층의 출발점이다([references/01-design-system.md](references/01-design-system.md)).

둘째, 데모의 핵심을 일찍 만들어 접근을 검증한다. 헤드리스 프리미티브를 차용해 오버레이·포커스 트랩·브레이크포인트 관찰을 손에 넣고([references/02-headless-primitives.md](references/02-headless-primitives.md)), 그 위에서 **적응형 시트를 가장 먼저** 조립한다. 시트 하나가 이후 모든 모바일 UX 패턴의 참조 구현이 되기 때문이다. 적응형 분기·safe-area·제스처·가상 스크롤이 모두 여기서 모인다([references/03-adaptive-ux.md](references/03-adaptive-ux.md)).

마지막으로 PWA로 마감한다. manifest와 서비스 워커, 설치 흐름을 얹어 앞서 만든 UX를 설치 가능한 앱으로 포장한다([references/04-pwa.md](references/04-pwa.md)).

구조에 관해서는 한 가지 절제를 지킨다. 처음에는 페이지 안에 두고, 같은 코드가 *실제로* 두 곳 이상에서 쓰일 때 비로소 하위 레이어로 추출한다. 빈 레이어를 미리 만들지 않는다.

## 변치 않는 원칙

몇 가지 원칙은 층을 가로질러 유지된다.

**Tailwind 클래스는 구현 디테일이며 공개 API가 아니다.** 디자인 시스템은 클래스 이름이 아니라 컴포넌트를 노출한다. 클래스를 밖으로 흘리는 순간 그것이 사실상의 API가 되어, 이후 내부를 바꾸기 어려워진다.

**접근성은 헤드리스 층에 둔다.** 포커스·키보드·ARIA·알림은 외형이 아니라 동작과 의미의 문제다. 동작 층에 두면 스타일을 아무리 바꿔도 접근성이 깨지지 않는다.

**열기·닫기 애니메이션은 transform과 opacity만 건드린다.** 레이아웃을 건드리지 않아야 컴포지터에서 부드럽게 돈다.

**제스처는 거리만이 아니라 속도까지 본다.** "얼마나 끌었나"와 함께 "얼마나 빨리 던졌나"로 닫을지 되돌릴지를 판정해야 손맛이 산다.

**SSR은 별개의 관심사다.** 이 스킬은 CSR을 전제한다. 다만 클라이언트 전용 동작은 렌더 시점을 명확히 두어(`afterNextRender` 등) 이후 SSR로 옮길 여지를 남긴다.

## 레퍼런스 구현 (이 레포)

- 토큰: `src/styles.css` (`@theme`, `[data-theme=light]`)
- 프리미티브: `src/shared/ui/` (Button · Sheet · ListItem · Snackbar · Checkbox)
- 적응형 기준점: `src/shared/lib/breakpoint.ts`
- 제스처 공통: `src/shared/lib/gsap.ts`
- 적응형 시트(참조 구현의 심장): `src/shared/ui/sheet/sheet.ts`
- 적응형 내비: `src/widgets/app-nav/app-nav.ts`
- 데이터(로컬 우선): `src/shared/api/` (Dexie + liveQuery)
- PWA: `ngsw-config.json` · `public/manifest.webmanifest` · `src/shared/lib/pwa-install.ts`
