# ADR-0009 · 프로젝트 구조

> 상태: ✅ 결정됨
> 일자: 2026-06-22
> 관련: [프로젝트 브리프 §3](../01.기획/프로젝트-브리프.md), [ADR-0005](./ADR-0005-데이터-계층.md)

## 1. 맥락

기능이 늘어도 의존 방향이 한쪽으로 흐르도록, 도메인·기능·UI를 계층으로 나누는 구조가 필요하다. 동시에 영화 카탈로그는 단순한 도메인이라, 처음부터 모든 레이어를 만들면 과설계다.

## 2. 검토한 대안

| 선택지 | 비고 |
|--------|------|
| 기능별 폴더(flat) | 공유·의존 방향 규율 약함 |
| 전통 레이어드(components/services/...) | 도메인 응집 약함 |
| **FSD v2.1 (최소 레이어 시작)** ✅ | 의존 방향 규율 + 점진 확장 |

## 3. 결정

**Feature-Sliced Design v2.1**을 채택하되, **최소 레이어로 시작**한다.

- 시작: **`app` · `pages` · `shared`** 3개로 출발.
- 실제로 두 곳 이상에서 재사용될 때 **`widgets` · `features` · `entities`**로 추출한다(미리 만들지 않음).
- **임포트 방향**: `app → pages → widgets → features → entities → shared`. 아래 레이어에서만 임포트하고, 같은 레이어 슬라이스끼리 직접 임포트 금지. 공개 API는 `index.ts`.

영화 도메인 슬라이스 예(추출 시점에):

```
src/
  app/                      # 부트스트랩, 프로바이더, 라우터
  pages/                    # home, movies, movie-detail, genre, search, wishlist, my-ratings, login
  widgets/                  # app-nav(적응형 네비)
  features/                 # add-to-wishlist, rate-movie, search-movies, share-movie
  entities/                 # movie, genre, rating, wishlist-item, session
  shared/                   # ui(AdaptiveSheet/Dialog, OverlayDismiss), lib(브레이크포인트·제스처·share), api(repository 추상), config(토큰)
```

- `shared/ui`: 순수 UI — 적응형 표현([ADR-0002](./ADR-0002-적응형-표현-전략.md)), 오버레이 닫기([ADR-0003](./ADR-0003-뒤로가기-오버레이-닫기.md)).
- `shared/api`: 저장소 Repository 추상([ADR-0005](./ADR-0005-데이터-계층.md)).
- `entities/session` + `AuthService`로 인증([ADR-0006](./ADR-0006-인증.md)).
- 기능 라우트는 **lazy loading**.

## 4. 주의사항

- 추출은 "두 번째 소비자가 생겼을 때"가 신호다 — 단일 소비자 단계에서는 페이지 안에 둔다.
- FSD 레이어를 Angular standalone/signals 컨벤션과 충돌 없이 매핑한다(NgModule 미사용).

## 5. 채택하지 않은 것

- **flat 기능 폴더**: 의존 방향 규율이 약하다.
- **처음부터 전체 6레이어**: 단순 도메인엔 과설계.

## 6. 결과

- 👍 의존 방향이 일관, 도메인 응집.
- 👍 최소 시작으로 초기 부담 없음, 점진 확장.
- 👎 추출 시점 판단과 공개 API 규율을 팀이 지켜야 함.

## 7. 후속 작업

- [ ] `app`/`pages`/`shared` 골격과 경로 별칭
- [ ] 임포트 방향 린트 규칙(가능하면)
- [ ] 재사용 발생 시 entities/features 추출
