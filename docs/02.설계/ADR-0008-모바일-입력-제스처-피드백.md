# ADR-0008 · 모바일 입력·제스처·피드백

> 상태: ✅ 결정됨
> 일자: 2026-06-22
> 관련: [모바일 인터랙션 리서치 §3.3~3.8](../01.기획/모바일-인터랙션-리서치.md), [ADR-0004](./ADR-0004-적응형-네비게이션.md)

## 1. 맥락

반응형 웹이 "앱처럼" 느껴지려면 레이아웃 외에 **입력·제스처·촉각/모션** 같은 모바일 고유 기대를 충족해야 한다. 이 ADR은 흩어지기 쉬운 이 결정들을 한곳에 모은다.

## 2. 검토한 대안

대부분 "직접 구현 vs 웹 표준 API 활용"의 선택이며, 프로젝트 원칙(단순한 것은 직접, 복잡한 동작은 검증된 것)에 따라 **표준 API/검증된 라이브러리**를 기본으로 한다.

## 3. 결정

| 영역 | 결정 | 구현 단서 |
|------|------|-----------|
| **가상 키보드 가림** | 입력칸이 키보드에 가리지 않게 보정 | `VisualViewport` API / `dvh` (필요 시 `VirtualKeyboard` + `env(keyboard-inset-*)`) |
| **입력 자동 줌(iOS)** | 포커스 시 줌 방지 | 입력 `font-size ≥ 16px` (`maximum-scale`로 막지 않음 — 접근성) |
| **키보드 최적화** | 입력 종류별 키패드 | `inputmode`(search/numeric 등), `autocomplete`, 검색은 `autocapitalize="off"` |
| **스와이프 액션** | 위시리스트 제거 등 | 임계값 통과로 확정 + **항상 보이는 대체 버튼**(접근성). 제스처 라이브러리는 [ADR-0001](./ADR-0001-ui-기술스택.md)에서 선정 |
| **호버 부재** | 호버 의존 금지 | `@media (hover:hover)`에서만 툴팁, 터치는 인라인/`:focus-within`, 탭 피드백 `:active` |
| **터치 타깃** | 충분한 크기 | 48px 목표, 간격 10px+ |
| **햅틱** | 확정 동작에 촉각 | `navigator.vibrate()` (모바일·사용자 제스처 한정) |
| **화면 전환** | 페이지 전환감 | View Transitions API, `@starting-style` + `transition-behavior: allow-discrete` |
| **safe-area** | 노치/홈 인디케이터 회피 | `viewport-fit=cover` + `env(safe-area-inset-*)` |
| **공유** | 영화 상세 공유 | `navigator.share()`(모바일 시트) + `canShare()` 분기, **데스크톱 클립보드 복사 fallback** |

## 4. 주의사항

- `navigator.share()`/`vibrate()`는 **사용자 제스처 + HTTPS**가 필요하다.
- 호버로만 드러나는 기능을 두지 않는다(터치에서 영영 안 보임).
- 입력 폰트 16px 규칙은 디자인 토큰에 반영해 전역 일관 적용.

## 5. 채택하지 않은 것

- **`maximum-scale=1`로 줌 차단**: 접근성 위배.
- **스와이프 전용 액션**: 키보드/스크린리더 사용자가 접근 불가 → 항상 대체 버튼 병행.
- **커스텀 제스처 엔진 직접 구현**: 엣지 케이스가 많아 검증된 라이브러리 우선.

## 6. 결과

- 👍 모바일에서 키보드·제스처·촉각·공유까지 앱에 가까운 경험.
- 👍 호버/포인터 분기로 데스크톱·터치 모두 깨지지 않음.
- 👎 표준 API 지원 편차(VirtualKeyboard·share)로 fallback을 항상 동반해야 함.

## 7. 후속 작업

- [ ] 입력 16px·inputmode·autocomplete 토큰/컨벤션
- [ ] VisualViewport/`dvh` 키보드 보정 유틸
- [ ] 스와이프 액션 + 대체 버튼 패턴(위시리스트)
- [ ] safe-area 토큰, View Transitions, 공유(share+복사) 유틸
