# 구현 계획

> 설계 문서가 정의한 구조를 어떤 순서로 구현할지 정합니다. 원칙은 두 가지입니다. 시각 토대(디자인 토큰)를 먼저 깔고, 데모의 핵심인 적응형 Sheet를 일찍 만들어 접근을 검증합니다. 그 위에 데이터와 기능을 얹습니다.
> FSD 원칙에 따라 처음에는 페이지에 두고, 재사용이 실제로 생길 때 하위 레이어로 추출합니다.

## Phase 0 — 기반

모든 시각 작업이 여기에 의존하므로 가장 먼저 깔아 둡니다.

- [ ] 경로 별칭 — `tsconfig.json`에 `@/*` → `src/*`
- [ ] 디자인 토큰 — `src/styles.css`의 `@theme`에 Discord 토큰(색·간격·라운드·브레이크포인트) 정의
- [ ] 대체 폰트 — Space Grotesk(디스플레이) + Inter(본문), Fontsource로 self-host
- [ ] CDK 설치 — `@angular/cdk`, `overlay-prebuilt.css` import
- [ ] `shared/ui`, `shared/lib` 골격 (첫 프리미티브를 만들 때 생성)

## Phase 1 — 디자인 시스템 프리미티브 + 적응형 Sheet

데모의 핵심입니다. Sheet 하나가 이후 모든 패턴의 참조 구현이 됩니다. 데이터는 아직 목업으로 둡니다.

- [ ] Button — 가장 단순한 프리미티브로 토큰 파이프라인을 검증합니다 (DESIGN.md variants)
- [ ] BreakpointObserver 래퍼(`shared/lib`) → signal로 노출
- [ ] 적응형 Sheet — CDK Overlay + BreakpointObserver + FocusTrap. 모바일 바텀시트 ↔ 데스크톱 모달, `safe-area-inset-bottom`, 끌어서 닫기 제스처, 열기/닫기 애니메이션

## Phase 2 — 데이터 레이어

UI가 검증되면 영속화를 붙입니다.

- [ ] 저장소 추상화 — Dexie DB 설정(`shared/api`) + localStorage 설정 서비스, 서비스 인터페이스로 캡슐화
- [ ] `navigator.storage.persist()` 요청
- [ ] Todo 모델 — 할 일 타입 + Dexie `liveQuery` → signal 연결

## Phase 3 — 첫 기능 수직 슬라이스: 할 일

프리미티브와 저장소를 실제 기능으로 엮는 첫 사례입니다.

- [ ] 홈 페이지 — 할 일 목록 (가상 스크롤)
- [ ] ListItem 프리미티브 (스와이프 가능)
- [ ] add-todo — Sheet의 첫 실사용처
- [ ] toggle-todo — 스와이프 완료 + 되돌리기 Snackbar
- [ ] 재사용이 확인되면 `entities/todo`로 추출

## Phase 4 — 나머지 기능

- [ ] 설정 — 테마 전환(다크 기본 ↔ 라이트, 토큰 스왑)
- [ ] 알림 — 리마인더, 권한 요청 흐름, 할일→알림 OHS 흐름
- [ ] 사용자 — 인증 (방식 확정 후)

## Phase 5 — PWA 셸

- [ ] `ng add @angular/pwa`, manifest, service worker, 오프라인 캐싱, 설치 흐름

## 진행 중 확정할 결정

- 제스처 라이브러리 선정 — Phase 1 Sheet 구현 시 (GSAP Draggable·InertiaPlugin / Interact.js 등 후보)
- 라이트 테마 팔레트 — Phase 4 설정 시
- 인증 방식과 데이터 시드·목업 전략 — Phase 3~4 사이
