# 모바일 인터랙션 리서치 — 웹 vs 모바일 사용자 경험

> 작성일: 2026-06-22
> 이 문서는 **반응형 웹에서 데스크톱과 모바일의 인터랙션 기대 차이**를 조사합니다. 기술 스택·라이브러리 선택은 [리서치](%EB%A6%AC%EC%84%9C%EC%B9%98.md), 저장소는 [데이터 저장 리서치](%EB%8D%B0%EC%9D%B4%ED%84%B0-%EC%A0%80%EC%9E%A5-%EB%A6%AC%EC%84%9C%EC%B9%98.md)를 참고하세요.
> 여기서 정리한 결론은 [`docs/02.설계/`](../02.설계/)의 ADR-0002·0003·0004·0007·0008로 이어집니다.

반응형 웹의 어려움은 화면 폭에 따라 **레이아웃(컬럼 수)** 을 바꾸는 데 있지 않습니다. 진짜 어려움은 같은 화면이라도 **사용자가 기대하는 동작이 환경마다 다르다**는 데 있습니다. 데스크톱 사용자는 모달과 호버에 익숙하고, 모바일 사용자는 뒤로가기 한 번으로 오버레이가 닫히고 풀스크린으로 작업하길 기대합니다. 이 문서는 그 차이를 세 갈래로 정리합니다.

## §1. 뒤로가기 ↔ 오버레이(다이얼로그·바텀시트·모달) 닫기

### 기대 동작

모바일 사용자, 특히 Android 사용자는 back 버튼/back 제스처를 **"한 단계 취소"** 로 인식합니다. 오버레이가 떠 있을 때 back을 누르면 **페이지가 뒤로 가는 게 아니라 오버레이가 닫혀야** 합니다.

- 모든 오버레이는 **두 가지 닫기 수단**을 제공해야 합니다: 명시적 Close 버튼 + 브라우저/OS의 back 버튼·back 제스처.
- 바텀시트는 swipe-down handle만으로는 닫는 법을 모르는 사용자가 많아, back으로 닫히지 않으면 길을 잃습니다(disorientation). back 지원이 이를 예방합니다.
- 단, **back으로 닫는 것 자체를 막지 말 것.** 꼭 필요한 동작이면 막는 대신, 같은 동작이 다시 필요할 때 오버레이를 재오픈하는 식으로 처리합니다.

### 표준 구현 ① — `history.pushState` + `popstate` (널리 쓰여온 패턴)

핵심은 "가짜 히스토리 엔트리"를 하나 만들어 back 버튼이 그 엔트리를 소비하게 하는 것입니다.

```javascript
// (1) 오버레이 열 때: 가짜 히스토리 엔트리 push
window.history.pushState('overlay-open', '');
window.addEventListener('popstate', closeOnBack);

// (2) back 누르면 popstate 발생 → 오버레이 닫기
function closeOnBack() {
  closeOverlay();
  window.removeEventListener('popstate', closeOnBack);
}

// (3) Close 버튼/Esc/배경 탭으로 직접 닫을 때: 가짜 엔트리를 직접 정리
function onManualClose() {
  closeOverlay();
  if (history.state === 'overlay-open') {
    window.history.back(); // 남은 가짜 엔트리 제거 → 히스토리 desync 방지
  }
}
```

> **가장 중요한 함정:** Close 버튼/Esc/배경 탭 등 **직접 닫는 경로**를 반드시 처리해야 합니다. 이때 `history.back()`으로 가짜 엔트리를 소비하지 않으면, 다음에 back을 눌렀을 때 이미 닫힌 오버레이를 닫는 popstate가 헛돌고 실제 페이지 이동이 한 번 어긋납니다.

### 표준 구현 ② — `CloseWatcher` API (history hack의 정식 대체재)

브라우저가 위 hack을 정식 API로 흡수했습니다. `CloseWatcher`는 Esc(데스크톱)와 **Android back 버튼**을 "close request"라는 단일 개념으로 통합합니다.

```javascript
const watcher = new CloseWatcher();
watcher.onclose = () => closeOverlay();             // back/Esc 시 호출
closeButton.onclick = () => watcher.requestClose(); // 수동 닫기도 동일 경로
// 저장 안 된 변경이 있으면 닫기 취소 가능:
watcher.addEventListener('cancel', e => { if (!saved) e.preventDefault(); });
```

- `requestClose()`는 `cancel`(취소 가능) → `close` 순으로, `close()`는 즉시, `destroy()`로 해제. close 후 watcher는 자동 파괴됩니다.
- 가장 최근 생성된 watcher만 이벤트를 받아 자연스러운 스택 동작을 제공합니다.
- abuse 방지: 사용자 활성화 없이 만든 여러 watcher는 하나로 묶여 한 번에 닫힙니다(back 버튼 무력화 방지).
- **`<dialog>`(showModal)과 popover는 이미 같은 인프라를 사용** → Android back으로 닫힙니다.
- **주의: 아직 Baseline 아님.** Chromium 계열 지원, Safari/Firefox는 미지원일 수 있습니다. 따라서 실무에서는 **CloseWatcher 우선 + history pushState fallback** 조합을 권장합니다.

> 참고: 순수 `<dialog>`는 기본적으로 Android back으로 닫히지 않고 히스토리를 이동하는 것이 기본이었습니다(이것이 CloseWatcher가 등장한 배경). `<dialog>`만 쓴다고 back-to-close가 공짜로 되는 건 아닙니다.

### route-based modal (라우트 기반 모달)

모달을 URL/라우트와 연결하는 접근입니다.

- **장점**: 모달 상태가 히스토리에 들어가 back으로 닫히고 forward로 재오픈됨, **딥링크·공유·북마크 가능**, 상태가 URL로 명료하게 표현됨.
- **단점**: 라우팅이 필요 없는 단순 확인/액션 모달까지 중첩 라우트로 만들면 과설계.
- → **공유가 필요한 화면(영화 상세)** 은 route-based modal로, **단순 확인/액션 시트**는 history/CloseWatcher로 — 용도에 따라 분리합니다.

### Angular 관점

- **CDK Overlay 자체엔 back-to-close가 없습니다.** 직접 디렉티브/서비스로 위 패턴을 붙여 추상화해야 합니다.
- Angular Material Dialog는 `closeOnNavigation`이 기본 `true`라 라우터 네비게이션 시 닫히지만, **이는 라우터 네비에만 반응**(CDK 커스텀 바텀시트엔 적용 안 됨). `canDeactivate` 가드와는 충돌 사례가 있음(가드 평가 전에 다이얼로그가 먼저 닫힘).
- **PWA에서는 Ionic의 `ionBackButton` 방식이 부적합**합니다(네이티브 Android에선 동작하나 PWA에선 미동작). PWA라면 History/CloseWatcher 패턴이 정답입니다.

### iOS Safari edge-swipe back 충돌

- iOS의 **화면 가장자리 가로 스와이프 = 뒤로가기**는 웹에서 쉽게 막을 수 없습니다. 모달/바텀시트가 자체 가로 스와이프(carousel, swipe-to-dismiss)를 쓰면 가장자리에서 시작된 제스처가 OS back과 충돌합니다.
- **권고**: 인터랙티브 영역을 화면 가장자리에서 떨어뜨리고, dismiss는 **vertical 스와이프 + back + 명시적 버튼**으로 다중화합니다.

### 이 프로젝트의 결론 → [ADR-0003](../02.설계/ADR-0003-%EB%92%A4%EB%A1%9C%EA%B0%80%EA%B8%B0-%EC%98%A4%EB%B2%84%EB%A0%88%EC%9D%B4-%EB%8B%AB%EA%B8%B0.md)

모든 오버레이는 back으로 닫히고 명시적 Close 버튼도 제공한다. 구현은 **CloseWatcher 우선 + history.pushState fallback**을 공용 디렉티브로 추상화한다. 공유 대상(영화 상세)은 route-based modal로 둔다.

## §2. 적응형 모달 ↔ 풀스크린/시트 (작은 화면 모달의 대안)

데스크톱에서는 모달(dialog)로, 모바일에서는 풀스크린 페이지나 바텀시트로 같은 콘텐츠를 보여주는 패턴입니다. 사용자가 말한 "모바일에서는 큰 페이지를 띄우고 싶다 / 스택"이 바로 이 패턴입니다.

### 정식 명칭들

- **Responsive Dialog / Adaptive modal** — 같은 콘텐츠를 뷰포트에 따라 다른 표현(modal vs full-screen vs sheet)으로 분기. shadcn/ui 생태계의 "Responsive Dialog".
- **Full-screen dialog (on mobile)** — Material Design의 공식 컴포넌트 명칭.
- **Route-based / intercepting modal** — 모달 상태를 URL에 묶어, 같은 URL이 데스크톱에서는 모달, 직접 접근/새로고침 시에는 전체 페이지로 렌더. **Instagram/Pinterest 사진 상세 패턴**의 웹 구현체.

### 분기 기준 — 디바이스가 아니라 viewport 폭 (Material Design 3)

- **Compact (폭 < 600dp, 세로 폰)** → **full-screen dialog**
- **Medium / Expanded (840dp~, 태블릿·데스크톱)** → **basic dialog**(scrim 위 modal window)
- full-screen dialog를 써야 하는 작업 기준: **키보드 입력이 필요**할 때, 변경이 **즉시 저장되지 않을** 때(명시적 저장/취소), 다이얼로그 안에서 **또 다른 다이얼로그**를 열 때.

### 핵심 설계 원칙

1. **콘텐츠는 한 번만 작성하고 표현 레이어만 스왑한다.** 데스크톱=중앙 모달(scrim), 모바일=풀스크린/바텀시트. shadcn Responsive Dialog(Vaul + Radix)가 깔끔한 웹 구현체 — 같은 콘텐츠 컴포넌트를 재사용하고 조건부 클래스로 표현만 바꿈.
2. **딥링크는 "콘텐츠에 항상 canonical URL을 두고, 모달은 그 URL을 가로채는 표현"** 으로 설계해야 유지된다. Next.js의 intercepting + parallel routes가 레퍼런스이며, 이것이 Instagram/Pinterest 동작 모델이다: 피드에서 클릭 → 모달(URL은 정식 경로) → 그 URL을 직접 열면 전체 페이지.

### 검증된 프레임워크는 어떻게 하나

| 프레임워크 | 접근 |
|-----------|------|
| **Ionic** | `ion-modal`의 `breakpoints`(예 `[0, 0.5, 1]`)로 바텀시트형 시트 모달(`0` 포함 시 아래로 스와이프 닫기). `presentingElement`에 라우터 아웃렛을 넘기면 **iOS 카드 모달**(뒤 페이지가 살짝 보임). `canDismiss`로 닫기 제어. **`ion-nav`는 push/pop 스택이지만 URL과 무관** → URL이 바뀌어야 하는 페이지 네비는 프레임워크 라우터를 써야 함 |
| **SwiftUI** | `NavigationStack` + `path` 바인딩. push = `path.append`, pop = `path.removeLast`. **데이터 주도**라 딥링크가 자연스러움(URL → path 배열 변환) |
| **Android** | Jetpack Navigation **back stack**. `navigate()`/`popBackStack()`. 딥링크 진입 시에도 상위 destination이 함께 쌓여 back이 자연스럽게 거슬러 올라감 |
| **Vaul + Radix (web)** | shadcn의 Responsive Dialog 권장 패턴: **데스크톱=Dialog, 모바일=Drawer**. 둘 다 Radix Dialog primitive 위라 접근성(focus trap·`aria-modal`·Esc) 내장, 표현만 미디어쿼리로 스왑 |

> Ionic의 "스택(push/pop)"은 종이를 쌓는 비유입니다 — 새 페이지를 위에 push(앞으로, back 버튼 생성)하고 맨 위를 pop(뒤로). 단 `ion-nav` 스택은 URL을 바꾸지 않으므로, **공유 가능한 페이지는 라우터로, URL 비연동 내부 스택만 ion-nav류로** 역할을 분리하는 것이 정석입니다.

### 모바일에서 데스크톱 모달이 나쁜 이유

- NN/g: **"큰 데스크톱 화면용 오버레이를 작은 화면에 띄우면 최선의 경우 혼란, 최악의 경우 콘텐츠 접근 자체를 차단한다."**
- **우발적 닫힘**: 모바일은 닫는 법이 제각각(버튼/바깥 탭/아래 스와이프/back)이라 잘못된 방법으로 의도보다 더 뒤로 가버리고 **작업이 날아간다.**
- Apple HIG: **"Minimize modality."** 시트=현재 맥락과 밀접한 **좁은 작업**(뒤 콘텐츠가 보여 맥락 유지), 풀스크린=**복잡·몰입형 작업**. 작은 화면에 떠 있는 전통적 데스크톱 모달은 권장하지 않음.
- 바텀시트 가이드(NN/g): back으로 닫기 지원, X 버튼 노출(grab handle만으론 불충분), **시트를 여러 겹 쌓지 말 것**, 페이지 네비게이션을 시트로 흉내내지 말 것 — 그건 풀스크린 페이지/스택의 역할.

### 이 프로젝트의 결론 → [ADR-0002](../02.설계/ADR-0002-%EC%A0%81%EC%9D%91%ED%98%95-%ED%91%9C%ED%98%84-%EC%A0%84%EB%9E%B5.md)

viewport 폭(CDK `BreakpointObserver`)으로 분기한다. **영화 상세 = 데스크톱 모달 / 모바일 풀스크린 페이지**, **평점 수정 = 데스크톱 모달 / 모바일 바텀시트**. 콘텐츠 컴포넌트는 한 번 작성하고 공용 `AdaptiveSheet`/`AdaptiveDialog`로 표현을 스왑한다. 영화 상세는 canonical URL(`/movies/:id`)을 항상 두고 목록에서의 진입만 모달로 가로챈다.

## §3. 웹 vs 모바일에서 사용자가 기대하는 그 밖의 동작 차이

레이아웃이 아니라 **인터랙션 기대치**의 대비입니다. 각 항목에 구현 단서(API/CSS)를 붙입니다.

### 3.1 네비게이션

| | 웹(데스크톱) 기대 | 모바일 기대 |
|---|---|---|
| 주 네비 | 상단 가로 네비/좌측 사이드바, 항상 보임 | **하단 탭바**(엄지 도달성), 3~5개 |
| 뒤로가기 | 브라우저 버튼 | **back 버튼 + iOS edge swipe** |

- **햄버거 메뉴**(NN/g 정량): 숨김 네비는 발견율 20%+ 하락, **데스크톱 과업 시간 39%↑**(모바일 ~15%↑). → **데스크톱에서 햄버거 지양**, 모바일은 4개 이하 노출/초과 시에만 햄버거. 핵심은 탭바 + 나머지는 메뉴인 하이브리드가 최선.
- 하단 탭은 메뉴/팝업을 열지 말고 **목적지로 직접 이동**.

### 3.2 스크롤 / 제스처

| | 웹 기대 | 모바일 기대 |
|---|---|---|
| 새로고침 | 버튼/F5 | **pull-to-refresh** |
| 항목 액션 | 호버 버튼/우클릭 | **스와이프**로 삭제/액션 |
| 스크롤 복원 | 보통 됨 | 뒤로가기 시 **이전 위치 복원 강하게 기대** |

- 구현: `overscroll-behavior: contain`(스크롤 체이닝/네이티브 pull-to-refresh 제어), `history.scrollRestoration`(`auto`/`manual` — SPA는 비동기 렌더 때문에 `manual` 직접 제어가 흔함).
- 스와이프 액션은 임계값(threshold) 통과로 확정하고, 접근성상 **항상 보이는 대체 버튼**을 함께 제공.

### 3.3 입력 (가상 키보드 / 줌 / 키보드 최적화)

| | 웹 기대 | 모바일 기대 |
|---|---|---|
| 키보드 | 하드웨어, 화면 안 가림 | **가상 키보드가 입력칸을 가림** |
| 포커스 | 줌 없음 | font-size < 16px면 **iOS 자동 줌인** |

- **키보드 가림**: layout viewport는 그대로 둔 채 visual viewport만 축소됨 → **`VisualViewport` API**로 가시 영역 추적, 또는 **`VirtualKeyboard` API**(`navigator.virtualKeyboard.overlaysContent = true` + `geometrychange` + `env(keyboard-inset-*)`), 간단히는 **`dvh`**.
- **자동 줌 방지**: 입력 `font-size: 16px` 이상(`maximum-scale=1`로 막는 건 접근성 위배라 비권장).
- **키보드 최적화**: `inputmode`(`numeric`/`decimal`/`email`/`search` 등), `autocomplete`, 이메일/검색엔 `autocapitalize="off"`.

### 3.4 오버레이 (바텀시트 / 툴팁 / 컨텍스트 메뉴)

| | 웹 기대 | 모바일 기대 |
|---|---|---|
| 보조 메뉴 | 드롭다운/팝오버(앵커에 붙음) | **바텀시트** |
| 정보 힌트 | **호버 툴팁** | 호버 없음 → 인라인/탭 |
| 컨텍스트 메뉴 | 우클릭 | **롱프레스** |

- 호버 툴팁은 `@media (hover: hover)`에서만, 터치는 인라인 또는 `:focus-within`(탭→포커스→표시).
- 바텀시트가 모바일에 맞는 진짜 이유는 도달성보다 **별도 페이지로 가지 않고 맥락을 유지**하기 때문(§2 참조).

### 3.5 터치 vs 포인터

| | 웹 기대 | 모바일 기대 |
|---|---|---|
| 정밀도 | 마우스 정밀 | 손가락 부정확 |
| 상태 피드백 | `:hover` | `:active`(탭), 호버 의존 금지 |
| 도달성 | 화면 어디든 | thumb zone(중앙/하단) |

- 터치 타깃 **48px 목표 · 간격 10px+**(Apple 44pt, Material 48dp, WCAG 2.5.8 44px).
- 입력 능력 분기: 데스크톱 `@media (hover: hover) and (pointer: fine)`, 터치 `@media (hover: none) and (pointer: coarse)`. 하이브리드(마우스 연결 태블릿)는 `any-hover`/`any-pointer`로 보정. **중요 기능을 호버 뒤에 숨기지 말 것.**

### 3.6 피드백 / 모션

| | 웹 기대 | 모바일 기대 |
|---|---|---|
| 촉각 | 없음 | **햅틱** |
| 화면 이동 | 즉시 교체 | **페이지 전환 애니메이션** |
| 화면 경계 | 직사각형 | 노치/홈 인디케이터 = **safe-area** |

- 햅틱: `navigator.vibrate()`(모바일 한정, **사용자 제스처 필요**) — 롱프레스 확정 등.
- 전환: **View Transitions API**(`document.startViewTransition()`), 진입/이탈은 `@starting-style` + `transition-behavior: allow-discrete`.
- safe-area: `<meta name="viewport" content="..., viewport-fit=cover">` + `env(safe-area-inset-*)`. 하단 탭바/바텀시트 패딩에 `env(safe-area-inset-bottom)`.

### 3.7 성능 / 오프라인

- 동일 Core Web Vitals 목표(LCP < 2.5s, INP < 200ms, CLS < 0.1)지만 **모바일에서 달성이 더 어려움** → 모바일/데스크톱 별도 모니터링, 실제 저사양 기기 + 실제 저속망 테스트.
- 오프라인 기대: Service Worker 캐싱 + **로컬 퍼스트(Dexie)** 로 "네트워크 때문에 작업을 잃지 않는다"는 기대 충족(이 프로젝트의 저장 전략과 직결).

### 3.8 공유 / 딥링크

| | 웹 기대 | 모바일 기대 |
|---|---|---|
| 공유 | URL 복사 → 붙여넣기 | **네이티브 공유 시트** |

- `navigator.share({title, text, url})`로 네이티브 공유 시트 호출. **HTTPS + 사용자 제스처 필요.** 데스크톱은 공유 UI가 없는 경우가 많으니 `navigator.canShare()`로 분기, **fallback은 클립보드 복사.**
