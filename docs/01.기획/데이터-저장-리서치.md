# 데이터 저장 리서치 — 프론트엔드 영속화

> 이 프로젝트는 백엔드를 범위에 두지 않습니다([프로젝트 브리프](%ED%94%84%EB%A1%9C%EC%A0%9D%ED%8A%B8-%EB%B8%8C%EB%A6%AC%ED%94%84.md)). 따라서 영화·위시리스트·평점·설정 데이터를 브라우저에 직접 저장해야 합니다.
> 이 문서는 브라우저 저장소 후보를 비교하고, 우리 데이터 특성에 맞는 전략을 추천합니다.

---

## 1. 후보 — 브라우저 저장소

브라우저가 제공하는 저장소는 성격이 뚜렷이 갈립니다.

| 저장소 | 모델 | 용량 | 접근 | 적합한 용도 |
| ------ | ---- | ---- | ---- | ----------- |
| **localStorage** | 문자열 key-value | 약 5 MB | 동기 (메인 스레드 블로킹) | 소량 설정·플래그 |
| **sessionStorage** | 문자열 key-value | 약 5 MB | 동기 | 탭 한정 임시 상태 (탭 닫으면 소멸) |
| **IndexedDB** | 구조화 객체 + 인덱스 + 트랜잭션 | 디스크 비율(큼) | 비동기 | 앱 데이터의 표준 |
| **Cache Storage** | 요청/응답 쌍 | 디스크 비율 | 비동기 | 서비스 워커의 자산·셸 캐시 (앱 데이터 아님) |
| **Cookies** | 작은 문자열 | 약 4 KB | 동기 | 서버 전송용 (앱 데이터에 부적합) |
| **OPFS + SQLite Wasm** | 파일 기반 SQL | 디스크 비율(큼) | 비동기 (워커) | 대용량·복잡한 SQL |

요약하면, **설정·세션처럼 작은 값은 localStorage**, **위시리스트·평점처럼 구조화되고 늘어나는 데이터는 IndexedDB**가 자연스러운 자리입니다. sessionStorage는 영속이 목적이 아니므로 데이터 저장소 후보에서 빠지고, Cache Storage는 PWA 셸 캐싱용이라 역할이 다릅니다.

---

## 2. 용량과 영속성 — 놓치기 쉬운 함정

저장소를 고를 때 용량보다 **데이터가 사라질 수 있다는 점**을 먼저 봐야 합니다.

- **localStorage**는 약 5 MB(UTF-16)에서 막히고, 초과하면 `QuotaExceededError`를 던집니다.
- **IndexedDB**는 용량을 브라우저가 동적으로 관리합니다. 기본은 "best-effort" 모드라, 기기 저장 공간이 부족하면 **데이터가 삭제(eviction)될 수 있습니다.**
- 이를 막으려면 `navigator.storage.persist()`로 "persistent" 모드를 요청합니다. 허용되면 그 출처의 데이터는 사용자가 직접 지우기 전까지 보존됩니다.
- `navigator.storage.estimate()`로 현재 사용량과 한도를 확인할 수 있습니다.

→ 위시리스트·평점처럼 잃으면 안 되는 데이터는 저장소만 고르는 것으로 끝나지 않습니다. **영속 권한을 함께 요청**해야 합니다.

---

## 3. 직접 구현 vs 라이브러리

IndexedDB는 강력하지만 저수준 API입니다. 트랜잭션, 커서, 스키마 버전 업그레이드를 직접 다루면 코드가 장황해지고 실수가 쉽습니다. 이 프로젝트의 원칙 — **단순한 것은 직접, 복잡한 동작은 검증된 라이브러리** — 을 그대로 적용하면, IndexedDB는 래퍼를 쓰는 편이 안전합니다([리서치 2절](%EB%A6%AC%EC%84%9C%EC%B9%98.md)의 제스처 판단과 같은 기준).

| 라이브러리 | 성격 | 적합 |
| ---------- | ---- | ---- |
| **Dexie.js** | 타입드 테이블, 인덱스 쿼리, 트랜잭션, 스키마 버전, `liveQuery`(반응형 읽기) | 앱 데이터의 기본 선택 |
| **localForage** | localStorage 같은 key-value API, IndexedDB 우선 + 폴백 | 단순 캐시·설정 |
| **idb** | 얇은 Promise 래퍼 | 브라우저 API를 거의 그대로 쓰고 싶을 때 |
| **RxDB** | 반응형 offline-first + 동기·충돌 해소 | 서버 동기가 생길 때 |
| **SQLite Wasm + OPFS** | 브라우저 내 풀 SQL | 복잡한 SQL·대용량 |

Dexie의 `liveQuery`는 데이터 변경을 구독 가능한 스트림으로 내보내므로, Angular 시그널과 자연스럽게 연결됩니다. 쿼리 결과가 바뀌면 화면이 따라 갱신되는 구조를 적은 코드로 만들 수 있습니다.

---

## 4. 우리 데이터의 특성

| 데이터 | 특성 | 맞는 저장소 |
| ------ | ---- | ----------- |
| 영화·장르 | 읽기 전용 목 데이터, 정렬·필터·검색 쿼리 | 정적 시드 → IndexedDB (Dexie) 적재 |
| 위시리스트·평점 | 구조화, 런타임 증가, 반응형 읽기 | IndexedDB (Dexie) |
| 설정·세션 | 소량 key-value, 동기 읽기가 편함 | localStorage |
| 임시 UI 상태 (시트/모달 열림 등) | 휘발성 | 메모리(시그널), 필요 시 sessionStorage |

---

## 5. 추천

- **영화·위시리스트·평점 → IndexedDB를 Dexie.js로 다룹니다.** 영화·장르는 정적 시드로 적재하고, 위시리스트·평점은 런타임에 쓰며 `liveQuery`를 시그널로 연결해 반응형 목록을 만듭니다.
- **설정·세션 → localStorage.** 양이 적고 동기 접근이 단순합니다.
- **핵심 데이터는 `navigator.storage.persist()`로 영속을 요청**하고, `estimate()`로 사용량을 모니터링합니다.
- **저장소 접근은 Angular 서비스로 추상화합니다.** 화면·컴포넌트는 저장소 구현을 모르고 서비스 인터페이스만 압니다. 이렇게 두면 나중에 원격 백엔드나 동기가 생겨도 교체가 쉽습니다([ADR-0005](../02.설계/ADR-0005-%EB%8D%B0%EC%9D%B4%ED%84%B0-%EA%B3%84%EC%B8%B5.md)).
- **지금 보류:** RxDB와 SQLite Wasm·OPFS는 현재 규모에 과합니다. 서버 동기, 대용량, 복잡한 SQL이 필요해지면 그때 재검토합니다.

---

## 6. 미확정

- 영속 권한(`persist()`)이 거부되었을 때의 UX
- 스키마 변경 시 마이그레이션 전략 (Dexie 버전 관리)
- 다중 탭 동기화 (`BroadcastChannel` 또는 Dexie의 변경 관찰)
